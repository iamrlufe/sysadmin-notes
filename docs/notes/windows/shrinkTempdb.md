# Сброс и уменьшение размера tempdb в MS SQL Server

Сбросить или уменьшить размер `tempdb` в MS SQL Server можно несколькими
способами. Выбор зависит от того, можно ли перезапустить службу SQL
Server прямо сейчас или проблему нужно решать на «живом» сервере без
остановки.

## Способ 1. Перезапуск службы SQL Server (самый надёжный)

При каждом перезапуске службы `tempdb` полностью пересоздаётся — все
файлы базы возвращаются к начальным (дефолтным) размерам, указанным в
настройках.

1. Откройте `services.msc` (Службы).
2. Найдите службу SQL Server (`MSSQLSERVER`) или именованный экземпляр.
3. Нажмите «Перезапустить» (Restart).

!!! warning "Разрыв соединений"

    Перезапуск службы кратковременно разрывает **все** активные
    соединения с сервером, а не только с `tempdb`. Планировать на окно
    обслуживания.

## Способ 2. Уменьшение размера «на лету» (без перезапуска)

Если перезапустить сервер сейчас нельзя, можно попробовать сжать файлы
базы вручную через T-SQL.

### Шаг 1. Очистка кэша (необязательно, но желательно)

SQL Server часто удерживает страницы `tempdb` в кэше, из-за чего
`DBCC SHRINKFILE` не даёт эффекта. Чтобы освободить память:

```sql
USE [tempdb];
GO
-- Запись незавершённых страниц на диск
CHECKPOINT;
GO
-- Очистка различных видов кэша
DBCC FREEPROCCACHE;
DBCC DROPCLEANBUFFERS;
DBCC FREESYSTEMCACHE ('ALL');
GO
```

!!! warning "Осторожно на проде"

    `DBCC FREEPROCCACHE` и `DBCC FREESYSTEMCACHE ('ALL')` — хороший
    вариант, если файл не сжимается, но на продакшен-сервере стоит быть
    аккуратнее: очистка процедурного кэша вызывает перекомпиляцию всех
    планов запросов, что может дать временный всплеск нагрузки на CPU.
    Если нужно почистить именно кэш временных таблиц, точнее и безопаснее
    выполнить:

    ```sql
    DBCC FREESYSTEMCACHE ('tempdb');
    GO
    ```

### Шаг 2. Сжатие файлов

Узнать логические имена файлов `tempdb`:

```sql
SELECT name, size/128.0 AS SizeInMB
FROM sys.master_files
WHERE database_id = DB_ID(N'tempdb');
```

Сжать нужный файл до желаемого размера (например, до 1024 МБ):

```sql
USE [tempdb];
GO
-- tempdev — стандартное логическое имя файла данных
DBCC SHRINKFILE (N'tempdev', 1024);
GO
-- templog — стандартное логическое имя файла журнала
DBCC SHRINKFILE (N'templog', 512);
GO
```

## Способ 3. Изменение начального (исходного) размера

Если `tempdb` изначально разрослась из-за неудачных настроек и после
перезапуска снова быстро вырастает до того же размера, стоит явно
задать меньший исходный размер файлов:

```sql
ALTER DATABASE [tempdb]
MODIFY FILE (NAME = N'tempdev', SIZE = 1024MB);
GO
ALTER DATABASE [tempdb]
MODIFY FILE (NAME = N'templog', SIZE = 512MB);
GO
```

Изменение вступит в силу при следующем перезапуске службы (`tempdb`
пересоздаётся с новым исходным размером).

## Диагностика: как найти «виновника» разрастания

Прежде чем сжимать файлы, полезно понять, что именно занимает место —
данные временных таблиц, системные объекты или версионность строк
(version store):

```sql
USE tempdb;
GO

SELECT
    SUM(user_object_reserved_page_count)*8/1024     AS UserObjects_MB,
    SUM(internal_object_reserved_page_count)*8/1024 AS InternalObjects_MB,
    SUM(version_store_reserved_page_count)*8/1024   AS VersionStore_MB,
    SUM(unallocated_extent_page_count)*8/1024       AS FreeSpace_MB
FROM sys.dm_db_file_space_usage;
```

Чаще всего причину разрастания стоит искать в неоптимизированных тяжёлых
запросах (большие `ORDER BY`/`GROUP BY`, использующие `tempdb`),
незакрытых транзакциях (растёт version store) или активном
использовании временных таблиц/табличных переменных.

## Особенности и подводные камни

### Несколько файлов данных tempdb

Начиная с SQL Server 2016 хорошей практикой считается несколько файлов
данных для `tempdb` (обычно 4 или 8, в зависимости от числа ядер CPU).
При работе с несколькими файлами:

- сжимайте (`DBCC SHRINKFILE`) или меняйте размер (`ALTER DATABASE ...
  MODIFY FILE`) у **всех** файлов данных одинаково;
- если файлы окажутся разного размера, SQL Server перестаёт
  использовать proportional fill (равномерное распределение записи по
  файлам), и нагрузка начинает уходить в основном на один файл.

### Авторасширение (Auto-Growth)

Убедитесь, что у всех файлов данных `tempdb` одинаковый начальный
размер и одинаковый шаг авторасширения, заданный в фиксированных МБ, а
не в процентах — иначе файлы будут расти неравномерно, что снова ломает
proportional fill.

### Флаг трассировки 1118 (для старых версий)

На SQL Server 2014 и старше стоит убедиться, что включены флаги
трассировки `-T1117` и `-T1118` — они предотвращают смешанные экстенты
и обеспечивают синхронный рост всех файлов `tempdb`. В SQL Server 2016+
это поведение включено по умолчанию, отдельно включать не нужно.
