# Автозагрузка программы на Windows Server (2016–2025)

Выбор способа зависит от того, должна ли программа запускаться только
при входе пользователя или сразу при включении сервера — без входа в
систему вообще.

## Способ 1. Папка «Автозагрузка» (запуск при входе пользователя)

Подходит, если приложение имеет графический интерфейс (GUI) и должно
запускаться после того, как вы вошли в систему — локально или через
RDP.

1. Нажмите `Win + R`.
2. Введите одну из команд:
    - `shell:startup` — автозапуск только для текущего пользователя;
    - `shell:common startup` — автозапуск для всех пользователей сервера.
3. В открывшуюся папку поместите ярлык программы (`.lnk`).

!!! warning "Работает только при активной сессии"

    Программа из этой папки запустится **только** когда указанный
    пользователь реально входит в систему интерактивно. Если RDP-сессия
    просто отключена (Disconnect), а не завершена (Log off), повторного
    запуска не будет — программа уже работает в той же сессии. Если
    сервер должен что-то запускать сразу после включения, без чьего-либо
    входа, — нужен способ 2 или 3.

## Способ 2. Планировщик заданий (рекомендуется для серверов)

Подходит, если программа должна работать постоянно в фоне, не требуя
входа пользователя при перезагрузке сервера.

1. `Win + R` → `taskschd.msc` → Enter.
2. В правой панели: «Создать задачу…» (Create Task).
3. Вкладка «Общие» (General):
    - укажите имя задачи;
    - отметьте «Выполнять для всех пользователей» / «Выполнять,
      независимо от того, выполнен ли вход пользователя»
      (Run whether user is logged on or not);
    - поставьте галочку «Выполнять с наивысшими правами» (Run with
      highest privileges) — **только если программе это действительно
      нужно**, см. предупреждение ниже.
4. Вкладка «Триггеры» (Triggers) → «Создать…» → в поле «Начать задачу»
   выберите «При запуске» (At startup) — для фонового сервиса без
   входа кого-либо, или «При входе в систему» (At log on) — если нужен
   именно вход конкретного пользователя.
5. Вкладка «Действия» (Actions) → «Создать…» → «Запуск программы»
   (Start a program) → укажите полный путь к `.exe`.
6. ОК → введите пароль администратора для сохранения задачи с опцией
   «независимо от входа пользователя» (пароль нужен, чтобы Windows могла
   запустить задачу без интерактивной сессии).

!!! tip "Задержка после старта"

    Триггер «При запуске» срабатывает очень рано — до того как
    полностью поднимется сеть или стартуют зависимые службы (СУБД, диск
    с сетевым путём и т.п.). Если программа падает именно при запуске
    сервера, но нормально работает при ручном перезапуске задачи —
    добавьте на вкладке «Триггеры» отложенный старт: «Отложить задачу
    на» (Delay task for) 30–60 секунд, либо явную зависимость на
    вкладке «Триггеры» → «При событии» от нужной службы.

!!! warning "«Наивысшие права» — не всегда"

    Галочка «Выполнять с наивысшими правами» требует, чтобы учётная
    запись, под которой выполняется задача, была администратором, и
    UAC-повышает процесс. Если программе не нужны права администратора,
    не отмечайте её — по принципу минимальных привилегий это снижает
    ущерб при компрометации самой программы.

На вкладке «Параметры» (Settings) стоит также включить «Перезапускать
при сбое каждые N минут» (If the task fails, restart every) и «Если
задача выполняется дольше — принудительно завершить» — обе опции
повышают отказоустойчивость без дополнительных скриптов.

## Способ 3. Превращение программы в службу Windows

Подходит для консольных приложений, скриптов и фоновых процессов,
которым нужна максимальная надёжность (автозапуск до входа кого-либо,
управление через `services.msc`, автоперезапуск при падении).

### Через встроенную утилиту `sc`

```cmd
sc create MyService binPath= "C:\Path\To\Program.exe" start= auto
```

!!! danger "Пробел после `binPath=` и `start=` обязателен"

    `sc` разбирает пары `ключ=значение` и требует ровно один пробел
    после знака `=`. `binPath="C:\..."` (без пробела) `sc` разберёт
    неверно и создаст сломанную службу без явной ошибки на этом шаге —
    проблема всплывёт только при попытке запуска.

Управление уже созданной службой:

```cmd
sc start MyService
sc stop MyService
sc description MyService "Что делает эта служба"
sc delete MyService
```

!!! warning "Не любой exe станет службой через `sc create`"

    `sc create` регистрирует программу как службу, но Windows ожидает,
    что исполняемый файл сам вызывает Service Control API
    (`StartServiceCtrlDispatcher` и обработку `SERVICE_CONTROL_*`).
    Обычная консольная программа или скрипт этого не делают — служба
    создастся, но при старте Windows выдаст ошибку вида «служба не
    ответила на запрос запуска или управления в течение установленного
    периода». Для таких программ нужен способ ниже.

### Через NSSM (Non-Sucking Service Manager)

Если программа не рассчитана на работу в качестве службы (обычный
`.exe`, `.bat`, PowerShell-скрипт) — NSSM оборачивает её так, что для
Service Control Manager она выглядит как полноценная служба.

```cmd
nssm install NameOfService
```

Команда без параметров откроет GUI: укажите путь к `.exe` (Path),
рабочий каталог (Startup directory) и аргументы (Arguments), затем
«Install service». Для установки без GUI (например, в скрипте
развёртывания) можно передать параметры сразу:

```cmd
nssm install NameOfService "C:\Path\To\Program.exe" "arg1 arg2"
```

Полезные последующие настройки:

```cmd
REM Автоперезапуск при падении процесса
nssm set NameOfService AppExit Default Restart

REM Перенаправление stdout/stderr в файлы (обычные консольные программы
REM иначе пишут вывод "в никуда")
nssm set NameOfService AppStdout C:\Logs\NameOfService.out.log
nssm set NameOfService AppStderr C:\Logs\NameOfService.err.log

REM Открыть GUI редактирования уже установленной службы
nssm edit NameOfService

REM Остановить и полностью удалить службу
nssm stop NameOfService
nssm remove NameOfService confirm
```

## Какой способ выбрать

| Способ | Нужен вход пользователя | Переживает перезагрузку без входа | Когда использовать |
|---|---|---|---|
| Папка «Автозагрузка» | Да | Нет | GUI-приложение, запускается вручную после входа по RDP |
| Планировщик заданий | Нет (при «Run whether user is logged on or not») | Да | Скрипты и программы без встроенной поддержки служб, нужна гибкость по триггерам |
| Служба Windows (`sc`/NSSM) | Нет | Да | Долгоживущий фоновый процесс, нужен автоперезапуск при падении и управление через `services.msc` |

## Особенности и подводные камни

- **Учётная запись задачи/службы.** По умолчанию служба, созданная
  через `sc create`, работает от `Local System` — очень широкие права.
  Если программе столько не нужно, задайте отдельную учётную запись:
  `sc config NameOfService obj= ".\ИмяПользователя" password= "..."`
  (или соответствующая вкладка «Log on» в NSSM / Планировщике заданий).
- **Путь с пробелами.** В `binPath=` и в поле «Программа/сценарий» на
  вкладке «Действия» путь с пробелами нужно брать в кавычки; если у
  программы есть свои аргументы — путь к `.exe` и аргументы разделяют
  на два поля («Программа» и «Аргументы»), а не собирают в одну строку.
- **`shell:startup` не масштабируется.** Для десятка серверов ярлыки в
  папке автозагрузки придётся раскладывать вручную или через GPO;
  Планировщик заданий и службы разворачиваются скриптом или через
  Group Policy Preferences / DSC без ручных действий на каждом сервере.
