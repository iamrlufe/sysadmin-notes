# Выгрузка бэкапов SQL Server на FTP через WinSCP (FULL/DIFF)

Когда SQL Server (например, через Ola Hallengren Maintenance Solution)
складывает бэкапы по папкам вида `<Корень>\<База>\<FULL|DIFF>\*.bak`,
после каждого бэкапа нужно выгрузить именно свежий файл на внешний
FTP-сервер — для хранения копии вне площадки. Три скрипта ниже решают
эту задачу: два тонких «ярлыка» (`upload_full.cmd`, `upload_diff.cmd`) и
общий «движок» (`upload_common.cmd`), который делает всю работу.

!!! warning "Про учётные данные"

    В скриптах ниже адрес FTP-сервера, логин и пароль заменены на
    плейсхолдеры `FTP_HOST` / `FTP_USER` / `FTP_PASSWORD` — в оригинале
    там были реальные значения открытым текстом. Ниже, в разделе
    «Особенности», — про то, почему так хранить пароль не стоит и что
    сделать вместо этого.

## Как это устроено

### Три файла, одна логика

`upload_full.cmd` и `upload_diff.cmd` — не более чем вызов
`upload_common.cmd` с параметром `FULL` или `DIFF` соответственно, плюс
`pause` в конце, чтобы окно не закрылось сразу и результат было видно
при ручном запуске. Удобно повесить их в Планировщик заданий Windows
сразу после соответствующих job'ов бэкапа. Вся логика — в
`upload_common.cmd`.

### Ожидаемая структура каталогов

```
E:\SQLBackup\
  <ИмяБазы1>\
    FULL\
      <ИмяБазы1>_full_...bak
    DIFF\
      <ИмяБазы1>_diff_...bak
  <ИмяБазы2>\
    FULL\
    DIFF\
```

Важно: скрипт узнаёт имя базы не из имени файла бэкапа, а из имени
папки на уровень выше `FULL`/`DIFF` — работает независимо от того, как
именно называются сами файлы `.bak`.

### Поиск нужных папок

`for /r "%ROOT%" %%D in (.) do` — приём cmd: `for /r` с шаблоном `.`
перебирает не файлы, а **подкаталоги** всего дерева `%ROOT%`, включая
сам корень. Для каждого каталога, чьё имя (`%%~nxD`) регистронезависимо
(`/I`) совпадает с `%TYPE%` (`FULL` или `DIFF`), вызывается подпрограмма
`:UPLOAD`.

### Определение базы и самого свежего файла

Внутри `:UPLOAD`: имя базы — это имя родительской папки
(`for %%I in ("%DIR%\..")`). Самый свежий `.bak` в папке ищется через
`dir /b /a-d /o-d` (без каталогов, сортировка по дате изменения — новые
сначала); из результата берётся только первая строка паттерном
«установить переменную, если она ещё не установлена»
(`if not defined FILE set "FILE=%%F"`).

### Защита от аплоада недописанного файла

Скрипт сравнивает размер файла до и после паузы в 5 секунд
(`timeout /t 5 /nobreak`). Если размер изменился — SQL Server ещё
дописывает бэкап, и скрипт выходит без выгрузки (при следующем запуске
попробует снова). Это эвристика, а не блокировка на уровне ОС: для
очень больших и медленно растущих бэкапов полностью исключить ложное
«файл готов» она не может, но для типичного случая этого достаточно.

### Сборка и запуск сценария WinSCP

Скрипт генерирует временный `.txt`-сценарий для `WinSCP.com`:

- `option batch continue` / `option confirm off` — не останавливаться и
  не спрашивать подтверждений на некритичных ошибках (например, если
  каталог на FTP уже существует);
- `open ftp://USER:PASS@HOST/ -rawsettings PassiveMode=1 FtpTLS=0
  SessionReconnectTimes=999 SessionReconnectTimeout=300` — подключение
  по обычному FTP (без TLS) в пассивном режиме, с почти неограниченным
  числом попыток переподключения — рассчитан на нестабильный канал;
- `lcd` в локальную папку с бэкапом, затем `mkdir`/`cd` в папку базы и
  `mkdir`/`cd` в `FULL`/`DIFF` на удалённой стороне — структура на FTP
  зеркалирует локальную;
- `put -resume "!FILE!"` — докачка, а не перезаливка с нуля, если
  соединение оборвалось на середине передачи;
- `exit`.

Сценарий запускается через `"%WINSCP%" /script=... /log=...`, результат
проверяется по `ERRORLEVEL`, лог сохраняется отдельным файлом на базу и
тип (`logs\<База>_<TYPE>.log`). Временный файл сценария удаляется в
конце независимо от результата.

## Как использовать

1. Установить WinSCP, проверить путь в `%WINSCP%`.
2. Прописать реальные `ROOT`, `HOST`, `USER`, `PASS` в
   `upload_common.cmd` (см. предупреждение про хранение пароля ниже).
3. Повесить `upload_full.cmd`/`upload_diff.cmd` в Планировщик заданий
   сразу после соответствующих job'ов бэкапа (или последним шагом самого
   job'а агента SQL Server).
4. После первого прогона проверить `logs\<База>_<TYPE>.log`.

## Полные скрипты

### upload_common.cmd

```bat
@echo off
setlocal EnableDelayedExpansion

REM ==================================================
REM upload_common.cmd
REM Вызывается так:
REM call upload_common.cmd "FULL"
REM call upload_common.cmd "DIFF"
REM ==================================================

set "TYPE=%~1"

if "%TYPE%"=="" (
    echo Не указан тип FULL или DIFF
    exit /b 1
)

REM ============================
REM НАСТРОЙКИ
REM ============================

set "ROOT=E:\SQLBackup"

set "WINSCP=C:\Program Files (x86)\WinSCP\WinSCP.com"

set "HOST=FTP_HOST"
set "USER=FTP_USER"
set "PASS=FTP_PASSWORD"

set "LOGDIR=%~dp0logs"

if not exist "%LOGDIR%" mkdir "%LOGDIR%"

echo.
echo ============================
echo Поиск %TYPE%
echo ============================

for /r "%ROOT%" %%D in (.) do (

    if /I "%%~nxD"=="%TYPE%" (

        call :UPLOAD "%%~fD"

    )

)

exit /b



:UPLOAD

set "DIR=%~1"

REM Имя базы = родительская папка

for %%I in ("%DIR%\..") do (
    set "DB=%%~nxI"
)

REM Ищем последний bak

set "FILE="

for /f "delims=" %%F in ('
dir "%DIR%\*.bak" /b /a-d /o-d 2^>nul
') do (
    if not defined FILE set "FILE=%%F"
)

if not defined FILE (
    echo.
    echo [%DB%] Нет файлов.
    exit /b
)

echo.
echo =====================================
echo База : %DB%
echo Тип  : %TYPE%
echo Файл : !FILE!
echo =====================================

REM --------------------------------------------------
REM Проверяем что файл уже не изменяется
REM --------------------------------------------------

set "SIZE1=0"
set "SIZE2=0"

for %%A in ("%DIR%\!FILE!") do set SIZE1=%%~zA

timeout /t 5 /nobreak >nul

for %%A in ("%DIR%\!FILE!") do set SIZE2=%%~zA

if NOT "%SIZE1%"=="%SIZE2%" (
    echo Файл еще записывается SQL Server.
    exit /b
)

set "SCRIPT=%TEMP%\winscp_%RANDOM%.txt"
set "LOG=%LOGDIR%\%DB%_%TYPE%.log"

(
echo option batch continue
echo option confirm off
echo option transfer binary

echo open ftp://%USER%:%PASS%@%HOST%/ -rawsettings PassiveMode=1 FtpTLS=0 SessionReconnectTimes=999 SessionReconnectTimeout=300

echo lcd "%DIR%"

REM Создаем каталог базы

echo mkdir "%DB%"
echo cd "%DB%"

REM Создаем FULL или DIFF

echo mkdir "%TYPE%"
echo cd "%TYPE%"

REM Загружаем файл

echo put -resume "!FILE!"

echo exit

)> "%SCRIPT%"

"%WINSCP%" /script="%SCRIPT%" /log="%LOG%" /loglevel=1

if ERRORLEVEL 1 (
    echo Ошибка загрузки.
) else (
    echo Успешно.
)

del "%SCRIPT%" >nul 2>&1

exit /b
```

### upload_full.cmd

```bat
@echo off
chcp 65001 >nul
setlocal

echo ==========================================
echo      SQL FULL Backup Upload
echo ==========================================
echo.

REM Запускаем общий скрипт для FULL

call "%~dp0upload_common.cmd" FULL

echo.
echo ==========================================
echo Готово.
echo ==========================================

pause
```

### upload_diff.cmd

```bat
@echo off
chcp 65001 >nul
setlocal

echo ==========================================
echo      SQL DIFF Backup Upload

echo.

REM Запускаем общий скрипт для DIFF

call "%~dp0upload_common.cmd" DIFF

echo.
echo ==========================================
echo Готово.
echo ==========================================

pause
```

## Особенности и подводные камни

- **Пароль в открытом виде в файле на диске.** `upload_common.cmd`
  хранит логин и пароль прямо в тексте скрипта — их видит любой, кто
  может прочитать файл. Лучше держать их отдельно от версии скрипта,
  которая куда-то расходится или лежит в общей папке: сохранённая сессия
  WinSCP (`winscp.ini` с мастер-паролем или интеграция с Windows
  Credential Manager через `/passive` и сохранённые сайты) либо
  переменные окружения, заданные отдельно, с ограниченными правами
  доступа на файл/ключ реестра, где они лежат.
- **FTP без шифрования (`FtpTLS=0`).** Пароль и содержимое бэкапа идут
  по сети открытым текстом. Там, где это возможно, стоит перейти на
  FTPS (`FtpTLS=1` или `-rawsettings FtpTLS=2` для implicit) или SFTP
  (`sftp://` вместо `ftp://`) — WinSCP поддерживает оба варианта теми же
  командами сценария.
- **Проверка «файл дописан» — эвристика.** Сравнение размера файла до и
  после 5-секундной паузы не гарантирует, что SQL Server закончил
  писать бэкап — для очень больших и медленно растущих файлов возможно
  ложное «файл готов». Надёжнее было бы проверять отсутствие открытых
  хендлов у файла или дожидаться завершения конкретного job'а бэкапа
  по расписанию, а не полагаться на побайтовую стабильность размера.
- **`mkdir` на существующую папку — не ошибка.** Благодаря
  `option batch continue` сценарий WinSCP просто продолжает работу,
  если удалённая папка базы или `FULL`/`DIFF` уже существует — это
  штатный сценарий при регулярных запусках, а не повод для тревоги в
  логе.
- **Пустая папка `FULL`/`DIFF` — тоже не ошибка.** Если файлов `.bak` в
  ней нет, `:UPLOAD` тихо завершается сообщением «Нет файлов» — обычный
  случай для новых или ещё ни разу не бэкапленных баз.
