# MikroTik как открытый DNS-резолвер (Allow Remote Requests) — причина и исправление

Классическое поведение RouterOS: если сеть внезапно «гонит» (канал и
процессор роутера забиты трафиком), а причина — включённая галочка
**Allow Remote Requests** в `IP → DNS`, дело почти наверняка в том, что
MikroTik превратился в открытый DNS-резолвер (Open DNS Resolver) и
используется для атак усиления. Ниже — почему так происходит и как
исправить, не потеряв DNS для локальной сети.

## Почему «гонит сеть» при включённой галочке

`Allow Remote Requests` заставляет MikroTik отвечать на DNS-запросы не
только из локальной сети (LAN), но и из всего внешнего интернета (WAN).
Открытые резолверы такого рода постоянно сканируются ботами для **DNS
Amplification Attack** — DDoS-атаки с усилением:

1. Боты шлют на ваш внешний IP короткие DNS-запросы с **подделанным**
   обратным адресом — реальным адресом жертвы атаки (source IP
   spoofing).
2. MikroTik честно отвечает на эти запросы — но ответ на некоторые типы
   DNS-запросов может быть в десятки раз больше самого запроса.
3. Раздутые ответы летят не ботам, а жертве, чей адрес был подставлен —
   но идут они **через ваш исходящий канал**, забивая upload и
   процессор роутера мусорным трафиком (сам по себе трафик вам не
   предназначен, но генерируете и оплачиваете его вы).

## Почему пропадает интернет, если галочку убрать

Если просто снять `Allow Remote Requests`, MikroTik вообще перестаёт
отвечать на DNS-запросы — в том числе от «своих» устройств локальной
сети. Если на компьютерах в LAN в качестве DNS-сервера указан IP самого
MikroTik (обычно `192.168.88.1`, что стандартно раздаётся DHCP-сервером
RouterOS по умолчанию), то без галочки роутер игнорирует и эти запросы
тоже — имена сайтов перестают резолвиться, и интернет визуально
«пропадает», хотя сама связность есть.

## Как проверить, что резолвер сейчас открыт

С любого внешнего хоста (не из вашей локальной сети — например, с
VPS или через мобильный интернет с телефона) выполните:

```bash
dig @<ваш_внешний_IP> google.com +short
```

Если приходит осмысленный ответ (IP-адреса) — резолвер отвечает на
внешние запросы и уязвим. Если приходит таймаут или отказ — снаружи он
не отвечает.

## Решение (2 варианта)

Оставлять галочку включённой без защиты — нельзя категорически.
Выберите один из двух вариантов.

### Вариант 1. Оставить галочку включённой, но закрыть доступ через Firewall (рекомендуется)

Правильный подход, если MikroTik должен работать как кеширующий
DNS-сервер для локальной сети.

1. Оставьте `Allow Remote Requests` включённой.
2. `IP → Firewall → Filter Rules`.
3. Добавьте два правила, блокирующих входящий DNS-трафик со стороны
   интернета — для UDP и для TCP:

    - Chain: `input`
    - In. Interface List: `WAN` (или список интерфейсов, куда
      подключён кабель провайдера)
    - Protocol: `udp` (вторым правилом — `tcp`)
    - Dst. Port: `53`
    - Action: `drop`

!!! danger "Порядок правил в Filter Rules критичен"

    Firewall в RouterOS обрабатывает правила сверху вниз и
    останавливается на первом совпадении. Перетащите эти два правила в
    **самый верх** списка `Filter Rules` — если выше них стоит
    разрешающее правило (например, широкое `accept` для уже
    установленных соединений, которое по ошибке пропускает и новый
    DNS-трафик), `drop` до него просто не дойдёт очередь.

### Вариант 2. Выключить галочку и раздавать DNS напрямую через DHCP

Подходит, если MikroTik вообще не должен обрабатывать DNS-запросы.

1. Уберите `Allow Remote Requests`.
2. `IP → DHCP Server → Networks`, откройте вашу локальную сеть.
3. В поле `DNS Servers` впишите публичные DNS вместо адреса роутера,
   например `8.8.8.8, 8.8.4.4`.
4. `Apply`.

После этого устройствам нужно переподключиться к сети (или обновить
IP по DHCP), чтобы начать слать DNS-запросы напрямую во внешний
сервис, минуя роутер.

## Особенности и подводные камни

- **IPv6 — отдельный firewall.** Правила выше закрывают только IPv4
  (`IP → Firewall`). Если на WAN включён IPv6, аналогичные правила
  (`drop`, `udp`/`tcp`, порт `53`, chain `input`) нужно продублировать
  в `IPv6 → Firewall → Filter Rules` — иначе резолвер останется открыт
  по IPv6 даже после закрытия IPv4.
- **Список интерфейсов `WAN` должен реально существовать.** Правило со
  значением `In. Interface List: WAN` сработает только если такой
  список создан в `Interface → Interface Lists` и в него добавлен
  реальный внешний интерфейс (или PPPoE-интерфейс подключения к
  провайдеру) — иначе правило либо не сработает вообще, либо потребует
  указать конкретный интерфейс напрямую вместо списка.
- **Вариант 1 не отменяет вариант 2 для гостевых/дополнительных сетей.**
  Если в сети несколько VLAN/интерфейсов с разными правилами доверия,
  для менее доверенных сегментов может быть проще сразу раздавать
  внешние DNS по DHCP (вариант 2), а `Allow Remote Requests` с
  firewall-защитой (вариант 1) оставить только для основной LAN.
- **Проверьте firewall уже после исправления.** Команда `dig` с
  внешнего хоста из раздела «Как проверить» — самый быстрый способ
  убедиться, что drop-правила реально отсекли внешние запросы, а не
  просто добавлены в список без эффекта из-за порядка правил.
