# Диагностика и управление агентами очереди Asterisk

Инструкция по диагностике статусов операторов, проверке SIP-регистрации, удалению и повторному добавлению агентов в очередь Asterisk.

---

## 1. Анализ статусов операторов

Для просмотра состояния очереди выполните:

```bash
queue show
```

Или для конкретной очереди:

```bash
queue show 131
```

### Статусы агентов

| Статус | Описание |
|---|---|
| `Not in use` | Всё хорошо. Агент активен, свободен и готов принимать звонки. |
| `Unavailable` | Проблема с аппаратом. Запись в очереди настроена верно, но телефон или софтфон недоступен. |
| `Invalid` | Ошибка конфигурации. Агент добавлен некорректно. Звонки на него идти не будут. |
| `In use` / `in call` | Агент сейчас разговаривает. |
| `Busy` | Устройство находится в состоянии «занято». |

> **Важно:** статус `Invalid` обычно указывает на некорректный интерфейс агента или невозможность Asterisk определить состояние указанного канала.

---

## 2. Проверка сетевого статуса аппаратов SIP

Чтобы понять, почему агент находится в статусе `Unavailable`, необходимо проверить регистрацию SIP-аппарата.

Для старого драйвера `chan_sip`:

```bash
sip show peers
```

Для современного драйвера `PJSIP`:

```bash
pjsip show endpoints
```

### Статусы SIP

| Статус | Описание |
|---|---|
| `OK (X ms)` | Телефон подключен к серверу и доступен. |
| `UNKNOWN` | Asterisk не может определить доступность устройства. |
| `(Unspecified)` | Телефон не зарегистрирован или не имеет актуального IP-адреса. |

Возможные причины:

- телефон физически отключен;
- отсутствует сеть;
- софтфон закрыт;
- изменились параметры авторизации;
- неверный пароль SIP;
- устройство потеряло регистрацию на сервере.

## 3. Как отключать агентов из очереди

Для удаления агента недостаточно всегда указывать только его внутренний номер.

Команда `queue remove member` требует указания **точного интерфейса агента**, отображаемого в выводе:

```bash
queue show 131
```

### Пример удаления некорректной записи

Если агент отображается так:

```text
1214 (ringinuse disabled) (dynamic) (Invalid)
```

Удаляем его командой:

```bash
queue remove member 1214 from 131
```

### Пример удаления агента Local-канала

Если агент отображается как:

```text
1219 (Local/1219@from-queue/n ...)
```

Необходимо использовать полный интерфейс:

```bash
queue remove member Local/1219@from-queue/n from 131
```

> **Важно:** интерфейс в команде удаления должен совпадать с интерфейсом агента в `queue show`.

## 4. Как правильно переподключать агентов

При ручном добавлении динамического агента необходимо правильно указать:

- интерфейс вызова;
- очередь;
- penalty;
- имя агента;
- интерфейс отслеживания состояния (`state_interface`).

Это позволяет Asterisk корректно определять:

- свободен ли оператор;
- разговаривает ли оператор;
- доступен ли внутренний номер.

### Эталонная команда

Пример для внутреннего номера `1219` и очереди `131`:

```bash
queue add member Local/1219@from-queue/n to 131 penalty 0 as 1219 state_interface hint:1219@ext-local
```

### Разбор параметров

#### `Local/1219@from-queue/n`

Диалплан-интерфейс, через который Asterisk выполняет вызов на внутренний номер.

```text
Local/1219@from-queue/n
```

Где:

- `Local` — технология канала;
- `1219` — внутренний номер;
- `from-queue` — контекст диалплана;
- `/n` — запрещает оптимизацию Local-канала.

---

#### `to 131`

Номер целевой очереди:

```text
to 131
```

В данном примере агент добавляется в очередь `131`.

---

#### `penalty 0`

Приоритет агента:

```text
penalty 0
```

Чем меньше значение `penalty`, тем выше приоритет агента.

Например:

```text
penalty 0
penalty 1
penalty 2
```

Агенты с `penalty 0` будут иметь более высокий приоритет.

---

#### `as 1219`

Отображаемое имя агента:

```text
as 1219
```

Используется для удобного отображения агента в очереди и логах.

---

#### `state_interface hint:1219@ext-local`

Интерфейс отслеживания состояния устройства:

```text
state_interface hint:1219@ext-local
```

Это один из наиболее важных параметров.

Asterisk использует `hint` внутреннего номера для определения состояния агента.

Благодаря этому статус агента может корректно изменяться между:

```text
Not in use
In use
Busy
Unavailable
```

Без корректного `state_interface` Asterisk может неправильно определять состояние агента.

---

## 5. Проверка агента после добавления

После добавления агента выполните:

```bash
queue show 131
```

Найдите необходимого оператора.

Корректный статус свободного агента:

```text
Not in use
```

Дополнительно проверьте SIP-регистрацию:

```bash
sip show peers
```

Для PJSIP:

```bash
pjsip show endpoints
```

Телефон должен быть зарегистрирован и доступен.

Для `chan_sip` обычно отображается:

```text
OK (X ms)
```

## 6. Что делать, если статус `Not in use`, но звонки не идут

Если:

- агент имеет статус `Not in use`;
- SIP-аппарат зарегистрирован;
- в `sip show peers` отображается `OK`;
- сотрудник не получает звонки;

необходимо выполнить дополнительную диагностику.

### 6.1. Проверка режима DND

Сотрудник мог случайно включить режим **Do Not Disturb**.

Для FreePBX код отключения DND по умолчанию:

```text
*79
```

Попросите сотрудника набрать:

```text
*79
```

> Feature Code может быть изменён администратором FreePBX. При необходимости проверьте текущие Feature Codes.

---

### 6.2. Проверка переадресации

Проверьте, не включена ли переадресация вызовов.

Коды отключения переадресации зависят от конфигурации FreePBX.

Проверьте:

```text
Admin → Feature Codes
```

И найдите параметры:

```text
Call Forward All Deactivate
Call Forward Busy Deactivate
Call Forward No Answer Deactivate
```

> Не следует считать `*74` универсальным кодом отмены всех переадресаций: Feature Codes могут отличаться в конкретной конфигурации FreePBX.

---

### 6.3. Принудительный тестовый звонок

Для проверки прохождения вызова напрямую до аппарата выполните в Asterisk CLI:

```bash
channel originate Local/1219@from-queue/n extension s@default
```

Если телефон зазвонил:

> Проблему необходимо искать в логике очереди, состоянии агента, паузе или настройках Queue.

Если телефон не зазвонил:

> Необходимо проверять диалплан, SIP-регистрацию, настройки аппарата и маршрутизацию вызова.

---

## 7. Быстрая последовательность диагностики

1. Проверить очередь:

```bash
queue show 131
```

2. Найти проблемного агента и определить его статус:

```text
Not in use
Unavailable
Invalid
In use
Busy
```

3. Проверить SIP-регистрацию:

```bash
sip show peers
```

Или:

```bash
pjsip show endpoints
```

4. Если агент имеет статус `Invalid` — удалить некорректную запись:

```bash
queue remove member <INTERFACE> from 131
```

5. Добавить агента заново:

```bash
queue add member Local/1219@from-queue/n to 131 penalty 0 as 1219 state_interface hint:1219@ext-local
```

6. Повторно проверить очередь:

```bash
queue show 131
```

7. Убедиться, что агент находится в статусе:

```text
Not in use
```

8. Если звонки не поступают — проверить DND, переадресацию и выполнить тестовый вызов.

---

## Пример полного восстановления агента

Удаляем старую запись:

```bash
queue remove member Local/1219@from-queue/n from 131
```

Добавляем агента заново:

```bash
queue add member Local/1219@from-queue/n to 131 penalty 0 as 1219 state_interface hint:1219@ext-local
```

Проверяем очередь:

```bash
queue show 131
```

Проверяем SIP:

```bash
sip show peers
```

Ожидаемый результат:

```text
Agent: Not in use
SIP: OK
```

Агент готов принимать звонки.
